# 第一轮组织蒸馏实验运行说明

本文档用于定义 **pmf-org-distill-skill** 的第一轮真实样本实验。当前这轮实验最重要的原则，不是要求用户提供“整理好的高质量数据”，而是尽量让系统能够接住**低门槛、粗糙、未经预处理的原始材料**。换句话说，用户只要能把自己手头已经有的工作痕迹放进来，系统就应该先把它们统一吸收，再逐步推断出组织中可能存在的接口问题、信息流堵点、决策链条断裂和角色协同摩擦。

这意味着我们的输入设计必须足够包容。它既要允许材料非常粗糙，例如一段口述困惑、几张群聊截图整理成文字、一次会议后的随手纪要、零散的任务分派记录，也要在材料丰富时继续提升效果，例如同时接入群聊、会议纪要、文档、工作流记录和阶段性复盘。**材料越丰富，分析效果通常越稳定；但材料不丰富，也必须能启动。**

## 一、实验目标

第一轮实验的核心目标有四个。第一，验证 `organizations/{slug}/evidence/` 下的真实粗糙材料能否被稳定接入。第二，验证不同模态和不同结构化程度的输入是否都能被归一化到统一分析单元。第三，验证 `normalized/`、`derived/` 与 `outputs/` 三层产物能否形成最小闭环。第四，验证用户在几乎不做预处理的情况下，是否也能得到一个可回溯、可解释的初步组织诊断结果。

| 验证对象 | 要回答的问题 | 通过标准 |
|---|---|---|
| 输入门槛 | 用户是否几乎不需要预处理就能提交材料 | 至少一种粗糙输入可直接进入 `evidence/` |
| 数据包容性 | 不同来源材料能否共存 | 口述、群聊、会议、文档、工作流至少可容纳其一 |
| 中间产物 | 标准化、线程、关系、假设能否形成链路 | `normalized/` 与 `derived/` 下能生成可检查产物 |
| 输出质量 | 最终诊断是否可回溯 | 输出中的关键判断能指出来源证据或线程 |
| 工程可维护性 | 目录、字段、文档是否一致 | README、架构文档、运行说明之间无明显冲突 |

## 二、实验目录

当前第一轮实验默认使用 `organizations/test-org/` 作为样本目录。目录骨架已经建立。用户不需要先做复杂标注，只需要把手头原始材料放进 `evidence/` 下最接近的类别中即可。

```text
organizations/test-org/
├── meta.json
├── evidence/
│   ├── docs/
│   ├── messages/
│   ├── meetings/
│   ├── decisions/
│   └── snapshots/
├── normalized/
├── derived/
├── outputs/
└── versions/
```

## 三、我们现在接受什么输入

这轮实验必须明确一件事：**输入不等于标准化后的知识库，而等于用户已经拥有的工作痕迹。** 因此，可接受的数据源应该尽量覆盖日常组织活动中最容易拿到的内容。

| 输入类型 | 推荐放置目录 | 是否允许粗糙原文 | 适合回答的问题 |
|---|---|---|---|
| 用户自述困惑或工作现状 | `evidence/docs/` | 允许，哪怕只是口语化文字 | 当前卡点是什么、日常协作怎么运转 |
| 连续群聊记录 | `evidence/messages/` | 允许，哪怕只有导出的文本或手工整理文本 | 谁在推动、谁在卡住、升级如何发生 |
| 会议纪要或转写 | `evidence/meetings/` | 允许，哪怕是简单纪要 | 共识如何形成、分歧如何暴露 |
| 决策记录或任务分派 | `evidence/decisions/` | 允许，哪怕只是几条任务列表 | 谁拍板、谁负责、谁被依赖 |
| 日常工作文档 | `evidence/docs/` | 允许，哪怕结构很乱 | 背景信息、上下游依赖、职责边界 |
| 工作流说明或 SOP 草稿 | `evidence/docs/` | 允许，哪怕是半成品 | 组织接口和正式流程是否与现实一致 |
| 截图、流程图、表格快照 | `evidence/snapshots/` | 允许 | 补充上下文或交叉验证 |

其中最重要的一类新增输入，是**用户自己的口述材料**。如果一个人没有现成的完整文档，也没有导出的聊天记录，他仍然可以写一段文字，讲清楚自己的岗位、最近的困惑、日常如何协作、最常见的卡点是什么、哪些人之间经常出现沟通断层。对系统来说，这类输入虽然主观，但仍然有很高的组织诊断价值，因为它至少提供了问题线索、角色集合和信息流假设。

## 四、最低可运行输入协议

为了真正降低门槛，我们不应该把“最小样本”定义得太苛刻。第一轮实验的**最低可运行输入**，建议定义为下表三档。只要满足任意一档，就可以开始跑。

| 档位 | 最低输入组合 | 适用场景 | 可得到的结果深度 |
|---|---|---|---|
| A 档 | 一份用户自述困惑文本 | 什么正式材料都没有时 | 可得到初步问题地图和待验证假设 |
| B 档 | 一份用户自述 + 一份群聊或会议纪要 | 有部分过程材料 | 可得到较可靠的线程与角色关系 |
| C 档 | 一份群聊/会议 + 一份决策记录/任务分派 | 有真实组织运行痕迹 | 可得到第一版接口问题与协作阻塞判断 |

这里的关键不是“材料必须多”，而是“材料至少能构成一个局部工作场景”。哪怕只有一个小问题的处理链，只要看得出参与人、讨论、动作和结果，就足够进入第一轮实验。

## 五、建议用户如何最省事地提供材料

如果我们想让更多人真正能用这个能力，就必须把提交方式设计得像“把已有东西扔进来”一样简单。下面这张表就是目前最推荐的提交方式。

| 用户手头有什么 | 建议怎么交 | 放到哪里 |
|---|---|---|
| 自己脑子里有困惑，但没整理过 | 直接写一段文字，像发长消息一样描述现状 | `evidence/docs/` |
| 微信/飞书/Slack 群聊 | 直接导出文本，或者复制粘贴成 txt/md | `evidence/messages/` |
| 会议后的简单纪要 | 原样放入，无需改格式 | `evidence/meetings/` |
| 任务分派、OKR、待办流转 | 原样放入 | `evidence/decisions/` |
| PRD、复盘、周报、排障记录 | 原样放入 | `evidence/docs/` |
| 流程图、截图、表格 | 直接放入 | `evidence/snapshots/` |

后续如果要进一步降低门槛，还可以支持一类特殊入口：**“组织诊断自述表”**。它本质上不是标准问卷，而是一份开放文本模板，让用户用自然语言回答几个问题，例如“你在哪个团队”“你最近最常见的协作问题是什么”“一个问题从提出到解决通常会经过谁”“你最常感到信息不透明的地方在哪里”。这种方式非常适合作为零门槛起点。

## 六、文件命名建议

文件名不要求特别严格，但建议尽量语义化，避免出现 `1.txt`、`新建文档.md`、`截图(3).png` 这种难以追踪的名字。

| 材料类型 | 推荐命名方式 | 示例 |
|---|---|---|
| 用户自述 | `日期_role_problem_statement.md` | `2026-04-11_ops_collab_confusion.md` |
| 群聊记录 | `日期_主题_渠道.txt` | `2026-04-10_release_blocker_wechat.txt` |
| 决策记录 | `日期_决策主题.md` | `2026-04-09_owner_assignment.md` |
| 文档材料 | `日期_文档类型_主题.md` | `2026-04-08_postmortem_api_delay.md` |
| 会议纪要 | `日期_会议主题.md` | `2026-04-07_weekly_sync.md` |

## 七、最小运行链路

第一轮实验默认按下面这条链路推进。前半段重点是把粗糙输入吸收到统一结构里，后半段才进入关系和问题推断。

| 阶段 | 输入 | 预期输出 | 目的 |
|---|---|---|---|
| 证据接入 | `evidence/` 下的原始材料 | 文件存在且分类清晰 | 确认输入准备完成 |
| 标准化 | 原始材料 | `normalized/evidence_units.json` | 把不同材料统一为可分析单元 |
| 线程重建 | 标准化结果 | `derived/thread_map.json` | 把连续消息、会议和问题叙述串成上下文 |
| 去噪评分 | 标准化结果与线程 | `derived/signal_scores.json` | 降低闲聊和弱信号干扰 |
| 关系映射 | 线程与评分 | `derived/relationship_map.json` | 抽出请求、汇报、拍板、阻塞等边 |
| 假设推断 | 关系与线程 | `derived/latent_hypotheses.json` | 形成保守的结构判断 |
| 最终输出 | 上述全部中间层 | `outputs/*.md` | 产出组织接口、信息流和问题诊断文档 |

## 八、第一轮验收标准

第一轮实验是否算“跑通”，不取决于结论多华丽，而取决于链路是否成立，以及是否真正体现了**低摩擦输入**这个原则。

| 验收项 | 合格标准 | 不合格表现 |
|---|---|---|
| 输入低门槛 | 用户几乎无需预处理即可提交一类材料 | 必须先人工整理半天才能开始 |
| 输入完整性 | 至少满足 A、B、C 三档中的任意一档 | 材料零散到无法形成局部场景 |
| 线程可读性 | 能识别主题、参与者和时间顺序 | 线程被切碎，无法追溯上下文 |
| 关系有效性 | 能识别至少一条拍板、升级、阻塞或请求边 | 只抽到空泛互动，没有结构含义 |
| 假设保守性 | 区分“已观察事实”和“推断假设” | 把猜测直接写成结论 |
| 输出可回溯性 | 最终判断能回指到中间产物或原始输入 | 结论悬空，找不到证据支撑 |

## 九、下一步协作方式

按照当前分工，接下来的配合方式应该继续保持简单。你负责把真实样本放入 `organizations/test-org/evidence/` 对应目录；我负责继续把运行入口、脚本衔接、字段修正、Prompt 收敛和实验检查补齐。

现在最建议你先放入下面任意一种最小组合。

| 组合 | 你要准备的材料 |
|---|---|
| 最省事组合 | 一份用户自述困惑文本 |
| 推荐组合 | 一份用户自述 + 一份连续群聊 |
| 最有实验价值组合 | 一份连续群聊 + 一份决策记录 + 一份补充文档 |

当你放好后，可以直接回一句：**“样本已放好”**。届时我就继续往下执行第一轮实验，并根据实际输入反向修正目录策略、字段设计和后续 Prompt。
