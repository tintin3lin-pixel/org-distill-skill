# Frontend Expression Guide

本文档用于规定 **pmf-org-distill-skill** 的前端表达层该怎么说。这里的“前端”不一定指网页界面，而是指**用户最终看到的那层话术和排版**。它必须做到三件事：第一，用户一眼就能看懂；第二，读起来有情绪张力，不像管理学教材；第三，再毒舌也不能脱离证据。

## 一、表达层的定位

这个产品的有趣之处，不在于把组织问题说得很学术，而在于把很多人早就模糊感觉到、却说不明白的东西，翻译成一句他们看了会点头、甚至会苦笑的判断。

> 用户不是来读论文的。用户是来确认：我到底是不是被这个组织结构坑了。

因此，前端表达层的职责不是“总结分析结果”，而是**把复杂判断压缩成一眼能懂的结论，再用少量解释把它稳住。**

## 二、整体语气

整体语气应该是：**清晰、锋利、轻量、有人味、适度毒舌，但不油腻。**

| 维度 | 应该是什么样 | 不应该是什么样 |
| --- | --- | --- |
| 清晰度 | 一句话能说清核心判断 | 写成大段空话 |
| 锋利度 | 能准确戳中用户处境 | 为了狠而狠 |
| 幽默感 | 像懂职场的人在说人话 | 像段子号在乱损人 |
| 专业性 | 每句毒舌都能回译成组织诊断语言 | 只有情绪，没有结构 |
| 轻量感 | 用户 30 秒能看懂重点 | 一上来就是长报告 |

## 三、推荐的最终呈现结构

建议最终用户可见版优先采用以下结构。长度不要太长，但层次必须清楚。

| 模块 | 作用 | 建议长度 |
| --- | --- | --- |
| 标题 | 用一句话点题 | 1 行 |
| 一句话判词 | 最核心的组织判断 | 1 行 |
| 你的位置 | 点出用户在系统里的角色 | 1 段短句 |
| 为什么会这样 | 解释背后的结构原因 | 2 至 4 段短文 |
| 留还是走 | 给出行动倾向 | 1 段短文 |
| 证据提醒 | 告诉用户这是基于哪些材料得出的 | 1 段短文 |

## 四、推荐标题风格

标题不应该像咨询报告，也不应该像营销海报。它更适合是那种**一看就知道在说组织病灶**的标题。

| 类型 | 示例 |
| --- | --- |
| 结构型 | `你的问题不在同事，在你们公司的信息批发系统` |
| 位置型 | `你以为自己是负责人，其实你只是故障缓冲层` |
| 决策型 | `这家公司不是没人干活，是没人真的拍板` |
| 去留型 | `这不是“再努力一下”能解决的局` |

## 五、一句话判词模板

一句话判词是整个产品最重要的钩子。它必须短、狠、准。

| 判断对象 | 模板示例 |
| --- | --- |
| 信息结构 | `这不是沟通问题，这是上下文只在少数人手里流通的问题。` |
| 权责结构 | `这不是分工清晰的团队，这是责任四处飘但结果要你接的团队。` |
| 用户位置 | `你现在不是在负责业务，你是在给组织结构擦屁股。` |
| 决策机制 | `这不是大家都在参与决策，这是没人愿意真正背决策责任。` |

## 六、“你的位置”模块模板

这个模块要帮助用户快速认出自己在系统里的功能位。不要写抽象人格评价，要写结构位置。

| 位置标签 | 说明话术 |
| --- | --- |
| `故障吸收垫` | `很多本该由系统解决的问题，最后都被你的人肉兜底了。` |
| `信息末梢` | `你承担结果，但拿不到完整上下文，所以你总是慢半拍。` |
| `中间缓冲层` | `你既没有完全的决定权，又不得不负责把上下游缝起来。` |
| `伪 owner` | `名义上你在负责，实际上关键资源、关键背景和关键拍板都不在你手里。` |
| `临时桥接点` | `组织没有稳定接口，所以很多沟通只能临时经过你。` |

## 七、“留还是走”模块模板

这个模块必须直接，但不能武断。建议始终分成三种结论口径。

| 结论 | 面向用户的话术 |
| --- | --- |
| `可以继续修` | `这局还没烂透。问题更像接口设计差，而不是系统性把人当耗材。你可以先争边界、争信息、争授权。` |
| `先观察别上头` | `这局现在还不值得你立刻掀桌，但也不值得你继续自我感动。先观察，再补证据。` |
| `建议准备撤退` | `如果你长期拿不到上下文、没有真实授权，却稳定承担结果，那你大概率不是在成长，是在被消耗。` |

## 八、表达层禁区

表达可以尖锐，但有几类话坚决不要说。

| 禁区 | 原因 |
| --- | --- |
| 直接骂某个人是废物 | 会把结构问题降格成情绪发泄 |
| 用绝对化语气替用户做人生决定 | 证据通常不足以支持这种断言 |
| 只给情绪价值，不解释结构 | 用户爽一下，但带不走任何洞察 |
| 过度玩梗 | 会削弱可信度，让产品像娱乐贴 |

## 九、推荐的最终用户版模板

```md
# 你的问题不只是某个同事不行

**一句话判词：** 这更像是一个上下文高度集中、责任分散外溢的组织系统，而你正在承担其中的缓冲成本。

## 你的位置
你现在更像一个“中间缓冲层”。很多事情不是你能拍板，但最后都要你去解释、去推进、去兜底。

## 为什么会这样
从你提供的聊天、会议纪要和任务分派来看，关键背景信息主要掌握在少数节点手里，但执行压力却被向下摊开。于是表面上每个人都参与协作，实际上真正知道全貌的人很少。这样的系统里，最容易累的不是最差的人，而是最愿意把事情接住的人。

第二个问题是接口虚化。很多任务虽然被分配了，但“谁有决定权”“谁负责提供背景”“谁负责最终兜底”并没有被稳定定义。于是你会不断感到自己像 owner，但又总在关键处卡住。这不是你不够强，而是你拿的是 owner 的责任，却不是 owner 的权限。

## 留还是走
现阶段更接近“先观察别上头”。这套系统有明显结构问题，但还需要看它是否允许边界修复。如果后续你继续长期承担结果、却拿不到更多上下文与授权，那就要开始认真准备退出，而不是继续靠责任心硬扛。

## 证据提醒
以上判断主要基于你提供的自述材料、与同事的对话片段、一次会议纪要以及任务分派记录。后续如果补充你和直属老板的互动材料，对判断会更稳。
```

## 十、最终原则

这个产品真正迷人的地方，不是“会骂人”，而是**它能把一句很有梗的话，落回一套很严谨的结构判断。** 用户读完应该产生的感觉不是“你替我骂爽了”，而是“原来我卡住，不只是因为我自己不行，而是因为我在一个有问题的信息结构里”。
