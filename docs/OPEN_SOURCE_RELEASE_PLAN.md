# pmf-org-distill-skill 开源发布评估与操作说明

## 一、先说结论

这套东西**适合开源发布，但不适合包装成“已经成熟的一键产品”**。更准确的定位应该是：它已经从概念层走到了**可跑真实样本的组织蒸馏 Skill 原型**，具备继续被别的开发者、别的 Agent 或研究型用户复用的价值；但它还没有达到“接到任何组织材料都能稳定输出高可信判断”的产品化成熟度。[1] [2]

因此，对外发布时最好的说法不是“我们已经做出了组织真相机器”，而是：

> **我们把一个能跑真实样本、能形成可回溯中间产物、能持续被真实反馈打磨的“组织反向蒸馏框架”开源了。**

## 二、真的能用吗

答案是**能用，但要带着边界去用**。

当前仓库已经具备一条真实存在的分析链：原始材料可以进入 `evidence/`，再被标准化成统一证据单元，随后进入线程重建、信号筛选、关系映射与隐性结构推断，最终落到可读输出层。这说明它已经不是纯概念草图，而是一个有中间层、有字段约定、有脚本入口的工程化原型。[1] [2]

但与此同时，文档里也明确承认当前状态是“**可继续编码，但还不是可交付产品**”。真实聊天导出格式、reply chain、跨文件线程合并、owner 链识别和置信度控制，都还处在继续补强阶段。[2] [3]

| 问题 | 结论 |
|---|---|
| 能不能跑 | 能跑 |
| 能不能用于真实样本试验 | 能 |
| 能不能作为稳定商用产品来卖 | 还不建议 |
| 能不能作为开源原型发布 | 可以，而且值得 |
| 结果能不能直接当终局真相 | 不能，应该当作保守诊断与辅助判断 |

## 三、它是不是别人的 Agent 也可以用的 Skill

答案是**可以复用，但还不是“零适配、通吃所有 Agent 平台”的即插即用商品”**。

如果一个 Agent 具备以下能力，它就已经可以使用这套仓库：读取 Markdown 说明、按目录组织输入材料、执行 Python 脚本、读取 JSON/Markdown 产物。对于这类 Agent 来说，这个项目已经是一个可移植的开源 Skill 原型。

但如果希望它直接进入某种通用 Skill 商店、无需适配即可被所有 Agent 平台识别，那么当前还差最后一段路。主要缺口包括：`SKILL.md` 的元信息与触发条件还可以继续标准化，Quickstart 示例还需要更明确，输出契约还应进一步统一，最好再补一份脱敏样例与平台适配说明。[1] [2] [3]

| 复用方式 | 当前是否成立 | 说明 |
|---|---|---|
| 人工按 README 手动试跑 | 成立 | 已具备目录协议与脚本链路 |
| 具备命令执行能力的 Agent 复用 | 成立 | 可以把仓库当作组织分析工作台 |
| 作为组织版 Skill 原型对外展示 | 成立 | 叙事清晰，差异化明显 |
| 作为通用 Skill 市场里的标准成品 | 暂未完全成立 | 还需补标准元信息、示例和适配说明 |

## 四、发布前最该改什么

从“能跑”到“适合公开讲”，最重要的不是继续堆更多判断，而是把对外叙事、使用入口和成熟度边界写清楚。

这次我已经完成两项关键文档改造。第一，已经重写仓库 `README.md`，把它从较内部的工程说明改成更适合公开传播的版本，重点强调三件事：它不是个人版同事/老板 Skill，而是**反向蒸馏组织**；它现在已经能跑真实样本，但仍属于**可复用原型**；它可以被别的 Agent 使用，但目前仍需要一定适配。第二，已经在公众号文章草稿结尾补上新的发布段落，把全文从“讨论同事.skill 引发的焦虑”收束到“我们为什么决定把组织蒸馏 Skill 开源出来”。

## 六、如果以后还要把别的仓库从 private 改成 public，怎么做

如果你未来还有别的同类仓库需要切成公开状态，可以用 GitHub 网页端或命令行两种方式。

| 方式 | 操作 |
|---|---|
| GitHub 网页端 | 进入仓库 → `Settings` → `General` → 拉到最下方 `Danger Zone` → `Change repository visibility` → 选择 `Change to public` |
| GitHub CLI | 在仓库目录执行 `gh repo edit --visibility public` |

如果用命令行，完整流程通常是：

```bash
cd /path/to/your/repo
gh repo view --json nameWithOwner,visibility,url
gh repo edit --visibility public
gh repo view --json nameWithOwner,visibility,url
```

公开之前，建议再做一次下面这张表里的检查。

| 发布前检查项 | 为什么要检查 |
|---|---|
| `.gitignore` 是否屏蔽真实样本与缓存 | 防止把敏感群聊、纪要或派生产物公开出去 |
| `README` 是否写清成熟度边界 | 避免外界误以为这是稳定商用产品 |
| 是否存在硬编码路径/个人信息 | 避免暴露本地环境与隐私细节 |
| 是否有脱敏样例或最小使用说明 | 让别人不是只看概念，而是能上手验证 |
| 文档口径是否一致 | 避免 README、SKILL、脚本输出互相打架 |

## 七、现在最建议的发布姿势

如果你准备正式宣传，我建议用下面这个叙事框架。

> **这不是一个“把某位同事做成 AI”的项目，而是一套把组织协作痕迹反向蒸馏为结构判断的开源 Skill 原型。**

它更像一个组织 CT 扫描器的早期工程版：不保证每一次都完美，但已经能从聊天、会议、文档和决策记录里，逐步还原出信息接口、关系链路、owner 缺口和权限结构。

这种说法的好处是，既有传播性，也足够诚实。它不会让人误以为这是一个已经成品化的神奇产品，但能清楚传达为什么这个方向值得被公开：因为在人人都在做“个人版 Skill”的时候，**把组织本身作为蒸馏对象**，是更稀缺、也更有解释力的一步。
