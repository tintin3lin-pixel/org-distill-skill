# FRAMEWORK_GAP_REVIEW

## 一、问题背景

这份评审文档用于解释：为什么当前版本的 **pmf-org-distill-skill** 容易产出“分析维度不够全面、结果偏表层、把闲聊误判为组织信号、无法识别群聊关系、也无法从显性材料反推出隐性组织结构”的结果。

问题并不主要出在写作表达，而是出在**分析框架本身还不完整**。当前骨架已经完成了从“识人”转向“识组织”的概念迁移，但还没有把“组织信息接口分析”真正落实成一套可执行的读取、过滤、建模与推断链路。

## 二、当前系统已经做对了什么

现有版本已经完成了几个重要的方向性改造。

| 已有能力 | 当前体现 |
|---|---|
| 从识人转向识组织 | `SKILL.md` 与 `ARCHITECTURE.md` 已把目标改写为角色接口、信息流与组织诊断 |
| 输出结构基本稳定 | 已有 `role_interfaces.md`、`info_flows.md`、`org_diagnosis.md` 三个核心输出 |
| 证据意识已经出现 | 文档已强调判断要尽量锚定材料来源，避免直接下人格结论 |
| 版本化骨架已存在 | `org_skill_writer.py`、`version_manager.py` 已具备最小闭环 |

这说明方向没有错，问题出在**中间层太薄**：从原材料进入结构化判断的那一层，还缺少足够强的分析机制。

## 三、当前最核心的缺口

### 1. 缺少“材料预处理层”

当前流程基本是“收材料 → 分三路分析 → 输出结果”。这意味着系统默认所有材料都已经是干净、可直接解释的证据。但现实不是这样。群聊、会议、邮件、文档里混杂着大量不同信号强度的内容，其中很多只是礼貌性寒暄、同步动作、情绪发泄或上下文缺失的碎片。

如果没有一个专门的**材料预处理层**，系统就会把“能看到的话”默认当成“值得解释的信号”。于是，闲聊被过度解读，片段被误当成结构，最后输出自然会显得浅。

### 2. 缺少“群聊关系理解层”

当前提示词会问“谁向谁输出什么”，但并没有要求系统真正重建一段群聊中的关系结构。例如，它没有系统地区分：谁在发起议题，谁在承接任务，谁在代老板转述，谁在缓冲冲突，谁只是围观附和。

也就是说，现在的系统更像是在读“句子”，而不是在读“关系”。而组织信息接口恰恰不主要藏在字面内容里，而是藏在：

> 谁有权跳过某些流程、谁必须等别人表态、谁的沉默也会形成约束、谁一说话就会触发收口。

这些都不是普通摘要能力能自动得到的，它需要专门的关系建模。

### 3. 缺少“信号过滤与分级机制”

虽然 `references/evidence_weighting.md` 已经写了证据权重，但它还停留在原则层，没有真正进入分析流程。当前代码里最明显的问题是 `evidence_indexer.py` 只按文件类别和文件数量做索引，并不理解内容，也不区分高价值线程和低价值闲聊。

于是系统当前其实只知道“有多少消息材料”，并不知道：

- 哪些是连续线程；
- 哪些是围绕一个议题的升级过程；
- 哪些只是插科打诨；
- 哪些对话虽然短，但暴露了权限边界；
- 哪些材料虽然长，但几乎没有组织信息含量。

这就导致“证据加权”在文档里存在，在实现里却几乎不存在。

### 4. 缺少“显性内容到隐性结构”的推断链

用户这次最关键的反馈，其实不是“分析不够多”，而是“没有剖析出深层的、没有在对话中直接体现的信息”。这说明当前系统仍然过于依赖显性表达。

但组织分析真正有价值的部分，往往恰恰来自**对未说出之物的结构性判断**。例如：

| 显性现象 | 背后可能对应的隐性结构 |
|---|---|
| 一线 repeatedly 追问背景，但始终没人解释 | 决策上下文没有被制度化下传 |
| 中层频繁改口，却从不解释依据 | 中层没有判断权，只有转述压力 |
| 群里所有人都等某个人最后拍板 | 形式上协作，实质上单点决策 |
| 老板只给结论不给过程 | 权限高度集中，组织以结果驱动替代认知对齐 |

现有版本虽然提到“信息损耗”和“权限边界”，但还没有把这种**从现象反推结构**的过程模块化。

### 5. 缺少“组织切片标注”之外的视角管理

当前文档已经提到，如果材料主要来自单一层级，应标注为“岗位切片画像”。这一步是对的，但还不够。因为真实问题不是只标注一句“这是切片”，而是要进一步说明：

- 这是从哪个位置看到的切片；
- 这个位置天然看不到什么；
- 当前偏差更可能来自信息不可见，而不是材料缺失；
- 还需要补哪一层的材料，才能判断当前推断是否成立。

也就是说，系统缺少一个**视角偏差管理层**。

## 四、当前 Prompt 层的问题

从现有 Prompt 来看，三个核心分析器都提出了正确方向，但都偏“提问式清单”，还不是“执行式工作流”。

| Prompt | 当前问题 |
|---|---|
| `role_interface_analyzer.md` | 会问接口问题，但没有要求先识别角色关系网络与发言权层级 |
| `info_flow_analyzer.md` | 会问流动路径，但没有要求先划定事件线程、议题起点与转述链 |
| `org_diagnosis_analyzer.md` | 会做诊断分层，但没有要求基于隐性结构假设来解释显性症状 |

换句话说，Prompt 还在问“你看到了什么”，但还没有要求系统按顺序完成“去噪 → 建模 → 推断 → 诊断”。

## 五、当前代码层的问题

当前代码层的问题比 Prompt 更明显，因为它几乎还没有承担真正的分析职责。

### 1. `evidence_indexer.py` 只做文件统计，不做内容理解

它当前只记录：路径、文件名、后缀、大小、修改时间。这对于仓库管理足够，但对于组织蒸馏远远不够。

它不知道一段群聊是不是同一个议题，不知道谁在和谁说话，也不知道某条消息是不是只是寒暄。因此它无法支撑“关系理解”和“闲聊过滤”。

### 2. 缺少 conversation parser / thread reconstructor

当前仓库没有任何脚本负责把群聊重建成“事件线程”。这意味着系统无法稳定回答下面这些组织分析中极关键的问题：

- 这段争论是谁先发起的；
- 谁在推动决策，谁在拖延决策；
- 谁在引用上级判断，谁在替上级背锅；
- 一次冲突到底经历了几个回合的升级。

没有线程重建，就很难做真正的组织接口分析。

### 3. 缺少 signal scorer / noise filter

当前没有模块负责给材料打“组织信息密度分”。因此所有材料在进入分析器前，基本处于平权状态。这与用户真正想要的结果是矛盾的。

因为组织蒸馏并不是“谁说得多就更重要”，而是“哪段材料更能暴露接口、权限和上下文损耗”。

### 4. 缺少 latent structure inference 层

系统现在没有一个明确模块，用来把显性材料转成隐性结构假设，例如：

- 决策权集中度；
- 解释权与执行权是否分离；
- 群体协作是否只是形式协作；
- 中层是否承担了过滤器而非判断者角色。

没有这一层，最终输出就只能停留在“现象总结”，而不是“结构诊断”。

## 六、为什么当前结果会显得“片面、表层、像在瞎分析”

归根到底，是因为当前系统缺的是一个完整的中间管道。现在的状态更像：

```text
原材料 -> 直接抽取若干组织关键词 -> 输出一个看似合理的结论
```

而用户真正想要的是：

```text
原材料
-> 去噪与切线程
-> 重建对话关系与发言权结构
-> 识别高价值组织信号
-> 从显性现象反推隐性组织结构
-> 再汇总成角色接口 / 信息流 / 组织诊断
```

这两者的深度不是同一个量级。前者更像写得好一点的摘要，后者才是真正的“组织蒸馏”。

## 七、下一版必须补上的模块

下一版如果要真正回应这次反馈，至少要把分析框架扩成下表这几层。

| 层级 | 必须补什么 | 作用 |
|---|---|---|
| Preprocessing Layer | 材料去噪、切线程、识别闲聊/信号 | 解决误读闲聊问题 |
| Relationship Layer | 群聊角色关系、议题发起者、承接者、拍板者识别 | 解决“不理解对话关系”问题 |
| Signal Layer | 组织信号强弱分级、证据权重动态化 | 解决“分析太片面”问题 |
| Inference Layer | 从显性材料反推隐性结构 | 解决“没有深层解读”问题 |
| Perspective Layer | 视角偏差与材料盲区说明 | 解决“只从单一位置误判全局”问题 |
| Synthesis Layer | 按组织信息接口汇总输出 | 保证结果真正围绕组织接口展开 |

## 八、结论

这次反馈说明，项目已经到了一个很明确的分水岭：如果继续沿用当前这种“材料归档 + 三个分析器”的轻骨架方式，输出会始终像一个组织主题的摘要器，而不是一个真正有洞察力的组织蒸馏器。

下一步的重点，不应该只是继续补更多 prompt 文案，而是要把“组织分析中间层”真正搭出来。只有这样，系统才不会继续把闲聊当信号、把表象当结构、把片段当结论，也才能真正回到这个项目最初的目的：**不是复述大家说了什么，而是看出这个组织到底是怎么运转的。**
