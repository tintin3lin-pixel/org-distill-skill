# 已知限制与常见误读

`pmf-org-distill-skill` 当前已经值得公开，但它最需要的不是夸张承诺，而是**边界说明**。这个文档的目标，就是把“它现在能做什么、不能做什么、最容易被误解成什么”集中讲清楚。[1] [2]

## 一、当前版本最重要的限制

| 限制 | 当前状态 | 为什么要明确写出来 |
| --- | --- | --- |
| 真实样本依赖仍然较强 | 公共仓库只能提供合成脱敏样例 | 公开样例无法完全替代真实组织复杂度 |
| 解释层仍偏原型 | 中间产物稳定度高于最终话术 | 避免外界误以为已是一键报告产品 |
| 平台无关性尚未完全验证 | 更适合具备脚本执行能力的 Agent | 防止在纯聊天平台上产生错误期待 |
| 一键入口尚不完整 | 当前更像工程原型而非终端产品 | 避免产品化预期错位 |
| 输出不是事实判决 | 仍需更多证据与人类判断交叉验证 | 防止把保守假设直接当裁决 |

## 二、最常见的四种误读

### 1. “读几段聊天就能判断整家公司”

这是一种典型误读。当前版本依赖的不是“几句有戏剧性的对白”，而是至少能构成一个局部工作场景的材料组合，例如用户自述加关键对话，或会议纪要加任务分派记录。[1] [3]

### 2. “这个项目是在给同事或老板做人格测评”

不是。这个项目的目标不是判断某个人天生是不是难搞，而是分析组织如何传递信息、如何形成决策、如何制造角色错位。个体只是结构的入口，不是全部解释。[1]

### 3. “输出结果可以直接拿去做 HR 或纪律处理”

不可以。当前输出是保守组织假设，适合复盘、研究、协作策略讨论和 Agent 工作流搭建，不适合做法律、HR、纪律处罚或心理诊断依据。[2]

### 4. “既然开源，就应该附真实样本”

这也是一种常见但危险的期待。这个项目处理的恰恰是最容易包含隐私、商业机密和上下文识别线索的材料，因此公开仓库必须以合成脱敏样例替代真实数据。[4]

## 三、推荐的对外自我介绍方式

如果要避免误解，最稳的介绍方式不是说“我们能看透组织”，而是说：

> 这是一个组织诊断骨架。它能把群聊、会议、任务分派、自述等工作痕迹转成结构化中间产物，并据此形成保守的组织假设；但它不是一键真相生成器，也不是纪律裁决工具。

## 四、最应该主动写出来的伦理边界

| 边界 | 建议口径 |
| --- | --- |
| 隐私边界 | 公共仓库不提交真实敏感材料，分享样例必须先脱敏或合成 |
| 解释边界 | 输出是结构化推断，不是不可推翻的事实 |
| 使用边界 | 适合研究、复盘、策略讨论，不适合处罚与定罪 |
| 自动化边界 | 当前最成熟的是分析骨架，不是一键终稿 |

## 五、对贡献者的提醒

如果你准备继续扩展这个项目，最值得坚持的一条原则是：**宁可把结论写得保守一点，也不要为了“像产品”而假装已经能稳定产出终局判断。** 这个项目最有价值的地方，恰恰在于它把模糊体感拆回证据与结构，而不是把不确定性重新包装成一种更顺滑的幻觉。[2] [3]
