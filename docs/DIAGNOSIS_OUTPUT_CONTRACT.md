# 组织诊断输出契约

本文档用于把 **pmf-org-distill-skill** 的输出层正式定义清楚：后端必须严谨，前端必须好懂。我们不是在做一份冷冰冰的组织管理报告，也不是在做一份纯情绪宣泄文本，而是在做一种**带证据约束的组织诊断表达**。

## 一、输出目标

这个 skill 的最终输出，至少要同时回答四个问题。第一，**这个组织的真实结构是什么样**。第二，**信息接口和决策接口卡在哪里**。第三，**用户在这个系统里到底处于什么位置**。第四，**从当前证据看，用户更适合继续熬、尝试修，还是尽快撤离**。

> 所有结论都必须尽量基于材料中的行为、语言、任务分派和上下文流动来推断，而不是仅凭用户情绪下判断。

## 二、双层输出原则

为了兼顾可信度和可传播性，最终输出必须拆成两层：**分析层** 与 **表达层**。

| 层级 | 目标 | 风格 | 约束 |
| --- | --- | --- | --- |
| 分析层 | 给出严谨、保守、可回溯的结构判断 | 冷静、清晰、可复核 | 每个关键判断都应能回指证据或中间产物 |
| 表达层 | 把分析结果翻译成用户一看就懂、愿意转发的说法 | 轻量、锋利、有梗、适度毒舌 | 不得脱离分析层胡乱发挥 |

这两层不能互相替代。**分析层负责可信度，表达层负责可感知度。** 如果只有分析层，产品会显得像咨询作业；如果只有表达层，产品会变成低质量嘴替。

## 三、分析层必须包含的模块

分析层建议至少包含以下模块。

| 模块 | 要回答的问题 | 输出要求 |
| --- | --- | --- |
| `org_structure` | 真实协作结构是什么样 | 识别核心推动者、拍板者、信息汇聚点、缓冲层、边缘角色 |
| `information_interfaces` | 信息如何流动、在哪里失真 | 识别高信息节点、低信息转述节点、阻塞节点和断裂接口 |
| `decision_mechanism` | 决策如何形成 | 判断是单点拍板、多头博弈、伪授权还是无人拍板 |
| `user_position` | 用户在局里扮演什么角色 | 判断用户更像 owner、执行器、补位者、协调层、背锅位或被隔离节点 |
| `structural_risks` | 当前系统的主要风险是什么 | 例如责任悬空、上下文丢失、频繁升级、角色错位、隐性权力大于正式权力 |
| `stay_leave_assessment` | 用户该留、该熬还是该撤 | 必须区分短期可修问题与长期结构性无解问题 |

## 四、表达层必须包含的模块

表达层不是把分析内容改写得更口语，而是要做一次**用户感知翻译**。建议至少输出以下模块。

| 模块 | 功能 | 示例方向 |
| --- | --- | --- |
| `one_line_verdict` | 一句话定性这个组织局面 | “这不是协作问题，这是上下文批发零售系统。” |
| `role_label` | 一句话点出用户当前位置 | “你现在不是 owner，你是组织故障吸收垫。” |
| `org_vibe` | 用一句更有画面感的话描述组织氛围 | “表面人人都在同步，实际上没人真的对全局负责。” |
| `stay_or_go` | 直接给出行动倾向 | “可以再观察，但别再免费当接口层。” |
| `sharp_explanations` | 用三到五段短文解释为什么 | 必须对应分析层的结构证据 |

## 五、去留判断的严格约束

“该留还是该走”是高敏感输出，不能写成鸡汤，也不能装作算命。这个模块必须明确区分三类结论。

| 结论 | 适用条件 | 说明 |
| --- | --- | --- |
| `stay_and_repair` | 问题集中在局部接口，且用户仍有上升空间或修复抓手 | 可以尝试调整边界、争取授权、改变协作方式 |
| `stay_and_observe` | 证据不足，或者系统混乱但暂未到不可逆程度 | 保守观察，继续补样本，不做过度承诺 |
| `leave_when_ready` | 结构问题反复出现，用户长期处于背锅位或信息末梢，且缺乏真实修复杠杆 | 建议尽早准备退出，不再对组织抱不切实际幻想 |

## 六、后端推断的理论口径

虽然产品前端可以有梗，但后端判断必须来自可解释框架。当前建议至少同时吸收以下几类视角，并在内部推断时保持保守：组织行为中的角色与权责错位、信息流与上下文损耗、社会网络中的中心性与桥接位置、协作心理中的归因偏差与风险转移。

这并不要求在前端把理论名词全部说出来，但要求内部分析能够分清：**什么是观察，什么是推断，什么只是待验证的怀疑。**

## 七、推荐输出文件

为了方便实验阶段检查，建议 `outputs/` 目录至少逐步形成以下文件。

| 文件 | 作用 |
| --- | --- |
| `analysis_report.md` | 严谨版分析输出 |
| `readable_brief.md` | 用户可直接阅读的轻量版输出 |
| `stay_leave_assessment.json` | 去留判断的结构化结果 |
| `evidence_trace.json` | 关键结论到证据单元的映射 |

## 八、最重要的边界

我们做的不是心理咨询，也不是职场情绪发泄器，更不是替用户做人生决策。这个 skill 的价值在于：**把“我怎么感觉哪都不对”翻译成“这个组织的问题到底出在结构、接口还是位置”，并把行动建议建立在可追溯证据之上。**
