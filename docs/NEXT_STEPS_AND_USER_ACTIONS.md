## 当前我已经补上的内容

这一轮我已经把 V2 最关键的**中间分析层骨架**先落下来了。项目现在不再只有“读材料 → 直接出结论”这一跳，而是新增了几层可继续深化的结构化处理：`tools/models.py` 统一了证据、线程、关系、评分和隐性假设的数据结构；`tools/evidence_normalizer.py` 负责把不同材料标准化成统一证据单元；`tools/thread_reconstructor.py` 负责把聊天、会议和决策材料重组成议题线程；`tools/noise_filter.py` 负责先做一轮启发式的去噪与信号标注；`tools/relationship_mapper.py` 负责把线程转成“谁向谁请求、汇报、升级、阻塞”的动作关系；`tools/latent_structure_inferer.py` 负责把显性结构进一步整理成可追溯的隐性组织假设。

与此同时，我也已经开始把提示词体系从“大一统分析”改成“分层协议化分析”，以避免系统再次把闲聊当结构、把表象当本质。

## 你现在唯一需要做好的事情

你当前最重要的工作，就是把真实材料放到正确的位置。推荐按如下结构准备：

```text
organizations/{slug}/
├── meta.json
├── evidence/
│   ├── docs/
│   ├── messages/
│   ├── meetings/
│   ├── decisions/
│   └── snapshots/
```

如果一开始材料还不全，也没关系。优先级最高的三类，是最能帮助我把 V2 做实的高信号材料。

| 优先级 | 材料 | 原因 |
| --- | --- | --- |
| 1 | `decisions/` 里的决策记录、任务分派、拍板信息 | 最容易识别权限结构与真正 owner |
| 2 | `docs/` 里的设计文档、评审意见、排障记录 | 最容易识别高信息接口和上下文持有者 |
| 3 | `messages/` 里的连续群聊片段 | 最容易识别请求、汇报、升级、催办和阻塞链路 |

## 你放文件时要遵守的约定

为了让现有代码后面可以直接接上，建议你在放材料时遵守下面这些约定。

| 约定 | 说明 |
| --- | --- |
| 文件名语义化 | 不要用 `1.txt`、`截图(2).png` 这种名字，尽量写清主题 |
| 尽量保留时间信息 | 群聊、会议、决策材料如果有时间戳，不要删 |
| 尽量保留说话人或作者 | 没有说话人，就很难做关系映射 |
| 连续片段优先于零碎截图 | 连续上下文更有利于线程重建 |
| 高信息材料优先 | 先放决策、评审、排障，再补闲聊背景 |
| 不要混放 | 文档、消息、会议、决策尽量按子目录分开 |

## 你给我反馈时，最有价值的是什么

等我这边继续推进实现后，你最有价值的反馈不是“感觉不对”，而是指出哪一种误判发生了。

| 反馈类型 | 示例 |
| --- | --- |
| 误把闲聊当组织信号 | 这段其实只是寒暄，不代表协作结构 |
| 没看出真实关系 | 这里其实是汇报链，不是平级讨论 |
| 没看出权限边界 | 这个人只是传话，不是真正拍板者 |
| 没看出隐性结构 | 真正的问题是 owner 缺失，但对话里没人明说 |
| 过度推断 | 这里证据不够，不能直接下这个结论 |
