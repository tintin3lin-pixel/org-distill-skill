# README 风格参考笔记

## 参考对象

| 仓库 | 观察到的开头风格 |
| --- | --- |
| tmstack/awesome-persona-skills | 一上来就用一句顺口、带挑衅感的话说明项目气质：`与其让别人蒸馏自己，不如抢先一步蒸馏别人`。句子短，读完立刻知道梗和态度。 |
| nicepkg/boss-skill | 先列几个打工人一眼能懂的具体场景，再接一句狠话：`他裁了你？没关系。AI 可以裁了他。` 情绪、对象、卖点都很集中。 |
| notdog1998/yourself-skill | 用对比句制造记忆点：`同事跑了，你蒸馏同事。前任分了，你蒸馏前任。` 然后迅速抛出自己的核心主张。 |

## 共同特征

这些 README 首页都有同一个特点：**不是先解释概念，而是先说人话。** 它们不会一上来解释架构、骨架、诊断框架，而是先讲一个大家已经懂的情绪和场景。

第二个共同点是，**句子很短，节奏很快。** 常见写法是两三句铺垫，然后马上给一句可以被截图传播的话。

第三个共同点是，**先交代“我们为什么搞这个”，再交代“它怎么工作”。** 也就是先有网感和立场，再有功能说明。

## 对当前 README 的改写启发

当前仓库首页的问题，不是方向不对，而是开头还在“解释项目”，没有先把读者拉进情绪里。下一版应该优先做到三件事。

第一，开头直接用打工人和组织关系里的真实感觉起手，不再抽象解释“组织分析骨架”。

第二，把核心句收敛成一句更好记的话，比如强调：**不是蒸馏一个人，是把整个组织蒸出来。**

第三，功能说明往后放，先让读者看懂“为什么会有人想要这个”。

## 用户点名参考：titanwings/colleague-skill

这份 README 的首页写法非常典型。它没有先解释什么叫 Skill，也没有先讲技术结构，而是连续抛出五个具体场景：同事离职、实习生离开、导师毕业、搭档调岗、前任交接不清。读者看到第二句时，已经知道这东西是在替谁说话。

然后它立刻用一句非常短的总结句收口：**Turn cold goodbyes into warm Skills**。这类句子有三个特点：短、顺、能被复述。

对当前仓库最直接的启发是：首页开头不要先解释“组织分析”这个概念，而要先说一个打工人能秒懂的现实场景，比如“你怀疑老板已经在找替代你的人，但你又说不清问题到底出在哪”。先把情绪拉出来，再说这个项目能把什么蒸出来。
