# ENGINEERING_GUIDELINES

## 一、文档目的

这份文档不是产品介绍，而是 **pmf-org-distill-skill** 的工程协作规范。它的目标很明确：无论后续是你自己继续写，还是交给另一个代码助手接手，都要尽量保证这个仓库在继续演化时，**命名一致、目录一致、接口一致、文档一致、代码质量判断标准一致**。

这个项目当前还处在骨架阶段，因此最怕的不是功能少，而是每个人都按自己的理解加一层，最后把整个结构写散。工程规范的意义，就是先把“哪些地方可以自由发挥，哪些地方必须严格统一”讲清楚。

## 二、项目的单一核心定义

从现在开始，这个仓库只有一个核心定义：它是一个**组织蒸馏 Skill 的工程骨架**，用于从原始材料中提炼组织的角色接口、信息流与权限结构。

这意味着任何新增功能、文档或脚本，都应该服务于下面这条主线，而不是偏离到“人格模拟器”“老板吐槽收集器”或“泛化知识库”上去。

| 层级 | 本项目的标准定义 | 不建议偏移成什么 |
|---|---|---|
| 产品目标 | 解释组织如何运转 | 模仿某个人说话 |
| 分析对象 | 角色接口、信息流、权限边界 | 性格标签、八卦印象 |
| 输入材料 | 文档、消息、会议、决策、截图等证据 | 纯主观评价、无上下文吐槽 |
| 输出结果 | 结构化组织分析 Skill | 单纯的人物画像 |

如果某个改动无法回答“它如何增强组织蒸馏能力”，就应该先暂停，不要直接并入主干。

## 三、必须统一的命名规则

这个仓库接下来最重要的基础约束之一，就是命名规则不能乱。命名一旦散掉，后面的 Prompt、Writer、Versioning、Collector 很快就会互相对不上。

### 1. 仓库与目录命名

仓库统一采用 `pmf-` 前缀，当前仓库名为 `pmf-org-distill-skill`。目录层级中，组织实例目录必须统一写到 `organizations/{slug}/` 下，不允许再平行引入 `teams/`、`companies/`、`cases/` 之类新的主目录，除非先同步修改架构文档。

### 2. 文件命名

核心输出文件必须固定为下表，不允许同义替换。

| 标准文件名 | 作用 | 不应替换成 |
|---|---|---|
| `role_interfaces.md` | 角色接口分析 | `roles.md`、`interfaces.md` |
| `info_flows.md` | 信息流分析 | `flows.md`、`information_flow.md` |
| `org_diagnosis.md` | 组织诊断 | `diagnosis.md`、`org_report.md` |
| `meta.json` | 元数据 | `metadata.json` |
| `evidence_index.json` | 证据索引 | `source_index.json` |

### 3. 脚本命名

工具脚本统一放在 `tools/` 下，命名使用小写加下划线。脚本名要直接反映职责，例如 `org_skill_writer.py`、`evidence_indexer.py`、`version_manager.py`。不要出现 `final_writer.py`、`new_tool.py`、`helper2.py` 这种无法表达职责的文件名。

### 4. Prompt 命名

Prompt 文件统一按功能命名，而不是按作者习惯命名。分析器类使用 `*_analyzer.md`，构建器类使用 `*_builder.md`，流程辅助类使用明确动作名，例如 `merger.md`、`correction_handler.md`。

## 四、目录变更的纪律

这个仓库当前的目录结构已经承担了“产品边界说明书”的角色。因此，目录不是想加就加，想改就改。后续如果要改目录，必须遵守一个基本原则：**目录变更必须先改文档，再改代码，最后补迁移说明**。

换句话说，任何新增一级目录、重命名关键文件、替换输出路径的行为，都不应该只改代码实现，而不更新 `README.md`、`ARCHITECTURE.md` 和这份规范文档。

推荐遵守下表中的变更顺序。

| 变更类型 | 正确顺序 |
|---|---|
| 新增核心目录 | 先更新 `ARCHITECTURE.md`，再改 README，最后写代码 |
| 重命名关键文件 | 先说明为什么改名，再同步修改所有引用与示例命令 |
| 增加新脚本 | 先定义职责边界，再决定是否应进入 `tools/` |
| 新增输出文件 | 先确认是否属于核心输出，而不是临时产物 |

## 五、文档一致性要求

从现在开始，项目中的四类核心文档要承担不同角色，不能互相抢位。

| 文档 | 角色定位 | 主要回答的问题 |
|---|---|---|
| `README.md` | 对外总览 | 这个项目是什么、怎么用、当前做到哪一步 |
| `ARCHITECTURE.md` | 设计说明 | 模块怎么分层、目录怎么组织、系统如何扩展 |
| `TODO.md` | 路线图 | 接下来按什么顺序做、优先级怎么排 |
| `ENGINEERING_GUIDELINES.md` | 工程规范 | 开发时哪些约束必须统一执行 |

文档之间必须满足以下一致性要求。

第一，**术语必须一致**。如果 README 里写“角色接口”，架构文档里就不要改成“岗位画像”，除非明确说明二者区别。

第二，**命令示例必须一致**。如果 `org_skill_writer.py` 的真实参数是 `--interfaces-patch`，任何文档都不能再写成别的名字。

第三，**状态描述必须一致**。如果某个脚本已经实现基础版，README、TODO 和架构文档里就不应该一处写“未开始”，另一处写“已完成”。

第四，**目录结构必须一致**。文档中的目录树要和仓库实际结构相匹配，不能长期保留已经不存在的占位结构。

## 六、代码质量底线

这个项目现在还不是成熟产品，但从骨架阶段开始就应该设定清晰的代码质量底线。这里说的“代码质量”，不是追求炫技，而是追求：别人接手时能看懂、能改动、能定位问题、不会越改越乱。

### 1. 基本要求

所有 Python 脚本至少满足以下条件：

| 维度 | 最低要求 |
|---|---|
| 可读性 | 变量名表达职责，不使用含糊缩写 |
| 可维护性 | 一个函数只做一类事情，避免过长函数 |
| 可运行性 | 提供最基本的命令行入口与参数说明 |
| 可诊断性 | 报错信息明确，不要无声失败 |
| 可扩展性 | 预留后续接 Prompt、Collector、Report 的接口空间 |

### 2. 不允许的坏味道

后续继续开发时，尽量避免以下问题进入主干：

- 把临时调试逻辑直接留在正式脚本里。
- 使用大量复制粘贴形成多个近似函数，却不抽公共逻辑。
- 在文档说一套、参数实现一套。
- 一个脚本同时处理写入、采集、分析、汇报四类职责。
- 使用 `final`、`new`、`temp`、`test2` 之类无语义命名。
- 只为了“先跑通”而写死路径、文件名或组织名。

### 3. 对骨架项目更重要的质量标准

由于当前阶段以脚手架为主，所以这里比算法聪明更重要的，是**边界清晰**。例如：写入器就负责写入，索引器就负责扫描，版本管理器就负责版本操作。谁都不要偷偷做别人的事。

## 七、脚本职责边界

为了避免后续代码助手把功能越写越混，当前建议对主要脚本做如下边界划分。

| 脚本 | 允许负责什么 | 不应该负责什么 |
|---|---|---|
| `org_skill_writer.py` | 创建、更新、列出组织 Skill，写核心输出文件 | 做复杂材料解析或自动抓取 |
| `evidence_indexer.py` | 扫描证据目录、统计来源、生成索引 | 直接下组织诊断结论 |
| `version_manager.py` | 列出、回滚、清理历史版本 | 改写核心分析内容 |
| `prompts/*.md` | 定义分析和生成模板 | 承担脚本行为逻辑 |
| `references/*.md` | 提供判断标准与参考分类 | 替代正式产品文档 |

如果后续要加飞书、钉钉或邮件采集器，也必须遵守同样原则：采集器负责“拿材料”，不负责“下判断”。

## 八、Prompt 与代码之间的契约

这个项目未来很可能会变成“Prompt + Python 工具 + 组织目录约定”协同工作的模式。因此，Prompt 与代码之间需要明确契约，而不是各写各的。

建议约定如下。

| 层级 | 约束 |
|---|---|
| Prompt 输入 | 使用统一术语，例如组织、角色接口、信息流、组织诊断 |
| Prompt 输出 | 尽量贴近目标文件结构，减少后处理成本 |
| Python 工具 | 只接收结构化输入或现成文件，不去猜测 Prompt 的随意输出 |
| 文档示例 | 必须使用真实存在的参数与路径 |

一个简单判断标准是：如果 Prompt 产出的结构，写入器要靠大量字符串猜测才能落盘，就说明契约还不够稳定。

## 九、提交与合并前检查

无论是人工开发还是交给另一个代码助手，建议每次提交前都至少过一遍下面这张检查表。

| 检查项 | 通过标准 |
|---|---|
| 命名检查 | 新增文件名是否符合现有命名规则 |
| 目录检查 | 是否引入了未在文档中说明的新目录 |
| 参数检查 | 文档里的命令示例是否能对应当前代码实现 |
| 状态检查 | README / ARCHITECTURE / TODO 是否需要同步更新 |
| 作用边界检查 | 新脚本是否只承担单一职责 |
| 基础运行检查 | 至少能执行帮助命令或基础示例，不出现明显报错 |

如果这张表里有两项以上没过，就不建议直接合并到主分支。

## 十、Definition of Done

这个仓库后续的任务完成标准，不应该只看“文件是不是写出来了”，而要看是否满足最小闭环。建议使用下面的完成定义。

| 任务类型 | 完成标准 |
|---|---|
| 文档更新 | 至少同步更新相关文档，且术语一致 |
| 新脚本加入 | 有清晰职责、有命令行入口、有最基本可运行性 |
| 参数变更 | README 与架构文档中的示例命令已同步更新 |
| 新目录引入 | 已说明用途、边界与后续维护方式 |
| 功能上线 | 至少经过一次样例目录验证 |

## 十一、给其他代码助手的接手原则

如果这个仓库接下来会交给另一个代码助手继续写，我建议它先遵守三件事。

第一，**先读文档，再改代码**。至少先通读 `README.md`、`ARCHITECTURE.md`、`TODO.md` 和这份规范文档，再决定从哪里下手。

第二，**先补全最小闭环，再加新能力**。也就是说，优先验证 create / update / index / rollback 这些基础动作，而不是一上来就铺飞书采集器或复杂分析器。

第三，**每改一个关键接口，就回头修正文档**。不要把文档更新留到最后，因为这个仓库当前最值钱的恰恰是“结构已被说清楚”。

## 十二、当前建议

从当前状态看，下一步最值得优先推进的并不是再发散功能，而是先做两件具体的事情。第一，用一个最小样例组织目录，把写入、更新、索引和回滚全部跑通一次。第二，把这次跑通过程里真实用到的命令、路径和注意事项，继续回写到 README 和 TODO。这样这个仓库才会从“文档完整的骨架”变成“可交接、可接力、可复用的工程起点”。

如果后续你确实准备让另一个代码助手接手，那这份文档就可以直接当成它的工程约束说明书来用。
