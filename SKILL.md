# org-distill-skill

## 目标

把组织材料整理成可复查的结构判断。重点看信息怎么传递、谁掌握上下文、谁在拍板、谁在补位，以及当前系统还有没有修复空间。

## 何时触发

出现下面这些情况时，优先触发本 Skill：

| 触发信号 | 说明 |
| --- | --- |
| 用户反复提到“团队很乱”“谁都说了算又谁都不拍板” | 适合分析真实决策路径与 owner 缺口 |
| 用户感觉自己一直在补位、兜底、背锅 | 适合分析角色位置与缓冲层风险 |
| 用户提供群聊、会议纪要、任务记录、周报、复盘等材料 | 适合进入证据化分析链路 |
| 用户想判断“该继续熬、修，还是准备离开” | 适合形成保守行动建议 |

出现下面这些情况时，不要直接触发，或只能部分触发：

| 不宜直接触发的情况 | 原因 |
| --- | --- |
| 只有纯情绪发泄，没有任何局部工作场景 | 证据不足，容易过度推断 |
| 用户想要法律、HR、纪律处罚结论 | 本 Skill 不提供此类裁决 |
| 用户只想做单人性格判断或心理测评 | 这不是本 Skill 的目标 |

## 接受的输入

本 Skill 接受未经精修的原始材料。用户不需要先做复杂整理，但材料至少要能构成一个局部工作场景。

| 输入类型 | 是否可单独使用 | 说明 |
| --- | --- | --- |
| 用户自述 | 可以 | 用来建立问题边界与用户位置初猜 |
| 私聊或群聊记录 | 可以 | 适合观察请求、汇报、升级、推责与上下文损耗 |
| 会议纪要或转写 | 可以 | 适合观察共识形成、冲突暴露与行动项分配 |
| 决策记录或任务分派 | 强烈推荐 | 对判断拍板路径与 owner 缺口价值最高 |
| 周报、复盘、SOP、流程文档 | 推荐 | 适合对比正式流程与真实流程 |
| 截图、流程图、白板快照 | 可选 | 作为辅助上下文 |

## Intake 最小字段

正式分析前，先整理出下面这份 JSON。不要先输出大段解释文字。

```json
{
  "name": "...",
  "scope": "...",
  "user_role": "...",
  "core_problem": "...",
  "stay_leave_question": "...",
  "suspected_pattern": "...",
  "interaction_targets": ["...", "..."],
  "available_sources": ["...", "..."]
}
```

字段含义如下：

| 字段 | 含义 |
| --- | --- |
| `name` | 本次侦察对象名称，例如某团队、某项目协作链 |
| `scope` | 本次分析边界，例如直属团队、跨部门项目组 |
| `user_role` | 用户所在位置，例如执行、协调、owner、接口层 |
| `core_problem` | 用户最想搞明白的问题 |
| `stay_leave_question` | 用户最关心的行动问题 |
| `suspected_pattern` | 用户主观怀疑的结构模式 |
| `interaction_targets` | 这轮材料里最关键的互动对象 |
| `available_sources` | 当前已有或准备提供的材料类型 |

## 最低可运行输入

| 档位 | 最低要求 | 可做的事 |
| --- | --- | --- |
| A | 一段用户自述 | 生成问题地图、角色假设与补样建议 |
| B | 用户自述 + 一段关键对话 | 初步判断接口摩擦、信息断层与用户位置 |
| C | 对话或会议 + 决策记录 | 较可靠地判断组织接口、拍板路径与结构风险 |

如果材料只有 A 档，不要强行给“留还是走”的重结论，先给保守假设和补证建议。

## 推荐流程

| 阶段 | 要做什么 | 输出 |
| --- | --- | --- |
| 1. Intake | 收束问题边界与输入类型 | 标准 intake JSON |
| 2. 证据整理 | 吸收原始材料，标准化为证据单元 | `evidence_units.json` |
| 3. 上下文重建 | 把证据串成线程与协作片段 | `thread_map.json` |
| 4. 信号判断 | 标记高价值片段与低价值噪音 | `signal_scores.json` |
| 5. 关系映射 | 识别请求、汇报、协调、阻塞、升级关系 | `relationship_map.json` |
| 6. 结构推断 | 形成保守的组织结构假设 | `latent_hypotheses.json` |
| 7. 表达收束 | 基于中间产物写成可读结论 | 报告、摘要或 Skill 文件 |

## 输出边界

输出时坚持下面这些原则：

| 原则 | 说明 |
| --- | --- |
| 输出是组织假设，不是事实判决 | 所有结论都应允许被进一步验证或修正 |
| 优先引用中间产物，不要只给情绪结论 | 保持可解释性与可复核性 |
| 行动建议必须区分“可修”与“结构性无解” | 不做空泛鼓励，也不装作算命 |
| 不得把结果用于法律、HR 或纪律处罚依据 | 明确伦理与使用边界 |

## 风格约束

表达可以直接，但不要拿情绪代替判断。每条结论都要能回到证据，能区分假设、观察和建议。

## 安装与接入要求

把这个仓库当成可安装 Skill 使用时，至少满足下面三件事：

| 要件 | 最低要求 |
| --- | --- |
| Skill 目录 | 保留一个包含 `SKILL.md` 的完整目录，不要只复制提示词片段 |
| 运行环境 | 能读取本地文件、执行 Python 3 命令、读写 JSON / Markdown |
| 数据接入 | 能把用户私有材料放入某个组织目录下的 `evidence/` 子目录 |

外部 Agent 应按目录协议和脚本链路接入，不要只复制提示词文本。

## 快速入口

首次接入时，按下面顺序阅读：

1. `README.md`：了解项目定位与边界。
2. `QUICKSTART.md`：按最小路径完成第一次试跑。
3. `samples/anonymized-minimal/`：使用可公开提交的脱敏样例。
4. `docs/STANDARD_OUTPUT_CONTRACT_EXAMPLE.md`：查看输出契约样例。
5. `docs/EXTERNAL_AGENT_ADAPTATION.md`：查看跨 Agent 接入方式。
